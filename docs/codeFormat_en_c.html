1. Naming Conventions

1.1 Variable Naming

  • Use camelCase for variable names (e.g., itIsAnExampleName).
  • Variable names should be clear and descriptive, avoiding abbreviations (except widely accepted ones like id, temp).
  • Keep local variable names concise and descriptive. For loop variables, single letters like i, j, etc., are acceptable, but maintain consistency and clarity, especially in nested loops.

1.2 Constant Naming

  • Use uppercase letters with underscores to separate words (e.g., MAX_BUFFER_SIZE).
  • Define constants with const or #define and place them at the beginning of the file or function definition to avoid "magic numbers" in the code.

1.3 Function Naming

  • Use camelCase for function names to maintain consistency with variable naming.
  • Function names should ideally include verbs to clarify their purpose, e.g., readFile(), calculateSum().

2. Commenting Guidelines

2.1 Comment Style

  • Use /* */ for both single-line and multi-line comments, avoiding //.

2.2 Comment Content

  • Add comments above definitions for variables or functions that arenâ€™t self-explanatory.
  • Each function should have a header comment that describes its purpose, input parameters, return values, and possible error handling.
  • Provide detailed comments for complex code blocks (e.g., algorithm implementations) to aid understanding.
  • Comment Example: c /* Calculates and returns the sum of two integers */ int calculateSum(int a, int b) { return a + b; }

2.3 Special Comments

  • TODO comments: Use /* TODO: describe task */ to mark incomplete code.
  • FIXME comments: Use /* FIXME: describe issue */ to mark known issues that need fixing.

3. Code Structure and Formatting

3.1 Indentation and Spacing

  • Use 4 spaces for indentation, avoiding tabs.
  • Leave spaces around operators (e.g., a + b).
  • Add a space after parentheses in functions, conditionals, and loops for readability.

3.2 Brace Placement

  • Place braces on a new line for all code blocks (e.g., if, for, while, switch).

  • Use braces for single-statement code blocks to improve consistency and maintainability.

    c if (condition) { /* code block */ }

3.3 Function Length Limit

  • Limit functions to 50 lines of code. If a function exceeds this limit, consider breaking it into smaller functions to improve readability and maintainability.

4. File Structure

4.1 Header Files

  • Include an include guard in each header file to prevent multiple inclusions. Use the following format: `c #ifndef FILENAMEH #define FILENAMEH

    / code /

    #endif /* FILENAMEH */ `

4.2 File Organization

  • Organize files by modules, with each module having its own file. Place common functions and variable definitions in separate header files.
  • .c files should contain implementation code only, while .h files should contain declarations and necessary comments.

4.3 Include Order

  • Include header files in the following order:
    1. Project-specific header files.
    2. Third-party library header files.
    3. Standard C library header files.

5. Error Handling

5.1 Error Return Values

  • Functions should handle all potential errors and return appropriate error codes (e.g., -1 or NULL), with each return condition documented in the function header comments.
  • For pointer-related operations, return a null pointer (NULL) to indicate errors.

5.2 Error Message Output

  • Use a consistent error output method (e.g., fprintf(stderr, ...)) and include detailed error information to assist in debugging.

6. Code Quality

6.1 Avoid Hardcoding

  • Define constants with const or #define to avoid "magic numbers" in the code.
  • Use custom data types or enumerations to enhance readability.

6.2 Memory Management

  • Free dynamically allocated memory when it is no longer needed to prevent memory leaks.
  • Avoid unnecessary dynamic memory allocation when possible; prefer stack memory where applicable.
